Gestión Crítica: Los Fallos en Bases de Datos

Autor: Estudiante de Ingeniería

Catedrático: Carlos Augusto Godínez Guzmán

Curso: Base de Datos 2

En el mundo de la ingeniería de datos, la disponibilidad y la integridad son pilares fundamentales. Sin embargo, ningun sistema es infalible. A continuacion, analizamos los fallos segun su origen y el impacto que tienen en SQL Server.

Nivel Lógico

1. Fallos de Transacción

Origen y Alcance: Estos fallos ocurren a nivel de unidad de trabajo logica. Su alcance suele limitarse a la transaccion actual, sin afectar la integridad global de la base de datos si se manejan correctamente.

Profundización: Pueden ser causados por errores de desbordamiento, violaciones de integridad referencial (llaves foraneas) o por el famoso "Deadlock" (bloqueo mutuo) donde dos procesos se esperan eternamente.

Manejo Profesional: La propiedad **ACID** (Atomicidad, Consistencia, Aislamiento y Durabilidad) es la que nos salva aqui. Usamos bloques `TRY...CATCH` para asegurar que el `ROLLBACK` limpie cualquier residuo de un intento fallido.

Ejemplo Real: Un sistema de inscripciones universitarias donde un estudiante intenta registrarse en un curso cupo lleno. El sistema intenta descontar el cupo, detecta que es 0, genera un error logico y debe revertir toda la operacion para no dejar datos inconsistentes.

BEGIN TRY
   BEGIN TRANSACTION;
   -- intento de transaccion
   UPDATE Cuentas SET Saldo = Saldo - 500 WHERE IdCuenta = 1;
   UPDATE Cuentas SET Saldo = Saldo + 500 WHERE IdCuenta = 2;
   COMMIT TRANSACTION;
END TRY
BEGIN CATCH
   ROLLBACK TRANSACTION;
   PRINT 'Error detectado: Transaccion revertida para mantener ACID.';
END CATCH;
Nivel Software/Energía

2. Fallos de Sistema (Crash)

Origen y Alcance: El origen es externo al motor de la base de datos, usualmente fallos en el Sistema Operativo, falta de memoria RAM o cortes de energia electrica. Su alcance afecta a todas las transacciones activas en memoria volatil.

Profundización: SQL Server utiliza un mecanismo llamado **Write-Ahead Logging (WAL)**. Esto significa que ningun dato se escribe en el archivo `.mdf` (datos) sin antes haber sido registrado en el `.ldf` (log). Al reiniciar tras un "crash", el motor realiza un proceso de *Recovery* automatico.

Ejemplo Real: Un servidor de un supermercado que procesa cientos de ventas por minuto sufre un apagon. Al volver la luz, el motor lee el Log de Transacciones y aplica las que estaban confirmadas (Redo) y deshace las que quedaron a medias (Undo).

Nivel Hardware

3. Fallos de Medio (Media Failure)

Origen y Alcance: Es el fallo mas temido. Su origen es el daño fisico de los platos del disco duro, falla en la controladora RAID o corrupcion de sectores donde reside la base de datos. Su alcance es total o parcial sobre los datos persistentes.

Profundización: Aqui la recuperacion no es automatica. Dependemos de la estrategia de Backup (Full, Differential, Log). La corrupcion de paginas de datos puede detectarse con el comando `DBCC CHECKDB`.

Ejemplo Real: El disco duro principal de un servidor contable hace un ruido extraño y deja de leerse. La unica solucion es reemplazar el hardware, montar el ultimo Backup Completo y aplicar los Backups de Log generados cada 15 minutos para minimizar la perdida de datos.

-- Restauracion profesional tras fallo de disco
RESTORE DATABASE MiBase FROM DISK = 'Z:\\Seguridad\\FullBackup.bak' WITH NORECOVERY;
RESTORE LOG MiBase FROM DISK = 'Z:\\Seguridad\\LogUltimo.trn' WITH RECOVERY;

4. Errores de Usuario y Operación

Nivel Humano

Origen y Alcance: El origen es el factor humano: negligencia, falta de pericia o fatiga. El alcance puede ser devastador, como borrar una tabla entera de produccion pensando que era la de pruebas.

Profundización: A diferencia de los otros fallos, el motor de la base de datos "cree" que la instruccion es valida (un `DELETE` sin `WHERE` es sintacticamente correcto). La prevencion se basa en el Principio de Menor Privilegio (PoLP).

Ejemplo Real: Un DBA junior ejecuta un `DROP TABLE Clientes` en el servidor de produccion por error. La recuperacion implica tecnicas de "Point-in-Time Recovery" para volver exactamente al segundo antes de que el usuario cometiera el error.
